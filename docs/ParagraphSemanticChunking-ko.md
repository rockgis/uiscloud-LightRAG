# 단락 시맨틱 청킹 전략 (Paragraph Semantic Chunking Strategy)

## 1. 사용 사례 및 전략 선택

### 1.1 P 전략이 해결하는 문제

단락 시맨틱 청킹 전략(이하 **P 전략**)은 DOCX와 같이 명확한 섹션 구조를 가진 문서를 대상으로 합니다. 핵심 목표는 토큰 길이 계산만으로 분할 지점을 결정하는 것이 아니라, **문서의 고유한 시맨틱 경계**(제목, 단락, 표 행)에 청크 경계를 최대한 맞추는 것입니다.

P 전략은 주로 다음 네 가지 범주의 문제를 해결하기 위해 설계되었습니다:

1. **표 컨텍스트 분절**: 대형 표가 분할될 때, 앞뒤의 설명 텍스트나 중간 연결 텍스트와 분리된 표 조각은 독립적으로 회상되면 이해하기 어렵습니다.
2. **계층 정보 활용 부족**: 인접 단락만 살피는 방법은 상위 제목 경로나 같은 절 사이의 관계를 활용할 수 없습니다.
3. **세분화된 절의 불균형한 크기**: 규정, 표준, 계약서 등에는 100–300 토큰의 세분화된 조항이 많습니다. 병합 없이는 청크가 너무 짧아지고; 인접 길이만으로 병합하면 토픽 오염이 발생합니다.
4. **긴 청크 재분할 시 구조 파괴**: 섹션이 지나치게 길면 일반 문자 분할은 표 행 경계와 제목 계층을 무시합니다.

P 전략은 `native` 추출 엔진이 생성한 `.blocks.jsonl` 구조화 아티팩트에만 효과적이며, 비구조화 입력에는 자동으로 R 전략으로 폴백합니다(§8 참조).

### 1.2 P / R / V 전략 비교

| 차원 | R 전략 (Recursive) | V 전략 (SemanticVector) | P 전략 (ParagraphSemantic) |
|---|---|---|---|
| 분할 기준 | 계단식 문자 구분자(단락 → 줄바꿈 → 중국어 구두점 → 공백 → 문자) + 토큰 한도 | 임베딩 거리 임계값(백분위수/표준편차/IQR/기울기)으로 시맨틱 분절점 탐지 | DOCX 개요 수준의 `parent_headings` + 표 행 경계 + 앵커 + 계층 인식 병합 |
| 청크 크기 제어 | `chunk_token_size` 하드 상한 | `chunk_token_size`는 권고 상한에 불과; 초과 시 R로 2차 분할 | `target_max` 하드 상한 + `target_ideal` 소프트 목표 + 표 임계값 + 꼬리 흡수 임계값 연동 |
| 표 처리 | 표 미인식; 표 중간에서 잘릴 수 있음 | 표 미인식 | `table_max` 미만의 표는 그대로 유지; 대형 표는 JSON 행 배열/HTML `<tr>` 행 경계로 슬라이스하여 유효한 `<table>`로 재포장 |
| 표 컨텍스트 | 우연한 윈도우 커버리지에 의존 | 임베딩 거리에 의존 | 첫 번째 슬라이스는 앞 설명 텍스트에, 마지막 슬라이스는 뒤 설명 텍스트에 결합; 연속 대형 표 사이의 연결 텍스트는 양방향 중복 |
| 청크 간 중복 | 전역 `chunk_overlap_token_size` | 중복 없음 | 섹션 경계를 넘는 중복 없음; 동일 섹션 내에서는 긴 본문이 R로 폴백 시 `CHUNK_P_OVERLAP_SIZE`로 중복; 연속 대형 표 사이의 연결 텍스트는 앞뒤 표 청크 모두에 들어갈 수 있음 |
| 제목 메타데이터 | 대체로 없음 | 대체로 없음 | 제목 상속 또는 승격; 분할 후 `[part n]` 접미사 추가; `parent_headings`와 `level` 보존 |
| 임베딩 계산 비용 | 없음 | 높음(문장별 임베딩 필요) | 없음 |
| 입력 요건 | 모든 텍스트 | 모든 텍스트 + 임베딩 모델 | `.blocks.jsonl` 사이드카 필요(즉, `native` 엔진 결과); 없으면 R로 폴백 |

### 1.3 전략 선택 방법

| 시나리오 | 권장 | 이유 |
|---|---|---|
| 명확한 섹션 계층, 대형 표, 세분화된 조항이 있는 DOCX | **P** | 제목 계층과 표 행 경계를 완전히 활용; 청크 경계가 시맨틱에 가장 잘 맞음; 토픽 오염 방지 |
| 명확한 섹션 구조 없이 산문/논평/긴 본문으로 구성된 문서 | **V** | 시맨틱 유사도로 분할하면 토픽 전환 지점에서 자연스러운 경계 형성 |
| 평문, 마크다운, 코드, 로그 입력이거나 최소 계산 오버헤드 원할 때 | **R** | 임베딩 오버헤드 없음; 계단식 구분자가 혼합 중국어-영어 텍스트에 충분히 안정적 |
| 일반 설정(파일 유형 불확실) | **R** | 사이드카 없을 때 P는 자동으로 R로 폴백; 임베딩 모델 없을 때 V도 R로 폴백 |
| 제목 스타일이 혼란스럽고 본문에 의사(pseudo) 제목이 많은 문서 | **R** 또는 **V** | P는 native 파서가 제목을 올바르게 식별해야 함; 혼란스러운 제목은 기본 청크 경계를 이동시킴 |
| 단일 행의 거대 표 또는 파싱 불가 표 | 어떤 것이든 | 세 전략 모두 결국 문자 수준 분할로 폴백; P는 여전히 표 컨텍스트 결합의 이점을 유지 |

### 1.4 P 전략의 비용

- `native` 엔진과 반드시 함께 사용: `LIGHTRAG_PARSER`에 명시적으로 선언 필요, 예: `docx:native-P`; 그렇지 않으면 `.blocks.jsonl`이 없어서 R로 폴백.
- DOCX 전용: 다른 형식에는 `.blocks.jsonl` 아티팩트가 없음.
- 많은 알고리즘 경로와 임계값: 디버깅 시 입력 사이드카를 먼저 검증한 후 각 단계 출력을 검사해야 함.

## 2. 작동 원리 개요

P 전략은 `fixlevel=0` 모드의 native 파서가 생성한 `.blocks.jsonl`을 입력으로 받습니다. **각 `type == "content"` 행을 하나의 제목 수준 기본 청크로 처리**한 후, 표 슬라이싱, 긴 청크 분할, 계층 병합을 수행합니다:

```text
DOCX
  ↓  native 파서 (fixlevel=0)
.blocks.jsonl + 사이드카 (.tables.json / .equations.json / .drawings.json / .blocks.assets/)
  ↓  Stage B: 행 경계를 따라 초과 크기 표를 슬라이스하고 first/middle/last 역할 부여
  ↓  Stage B.1: 연속 대형 표 사이 연결 텍스트의 양방향 중복
  ↓  Stage C: 앵커 기반 긴 텍스트 청크 재분할
  ↓  Stage D: 계층 인식 2단계 병합
  ↓  Stage E: [part n] 행 수준 출처 번호 부여
최종 청크 목록
```

**P 전략의 핵심 불변 조건**:

1. **섹션 경계를 넘는 중복 없음**: 서로 다른 `.blocks.jsonl` 내용 행 사이의 텍스트는 다른 청크에 복사되지 않아 "잘못된 귀속"을 방지합니다.
2. **섹션 내 긴 본문은 중복 가능**: 동일 내용 행에서 나온 여러 슬라이스는 `chunk_overlap_token_size`로 제어되는 R 스타일 중복을 유지할 수 있어, 긴 본문에서의 문장 중간 절단을 줄입니다.
3. **표 사이 연결 텍스트는 양방향 중복 가능**: 유일하게 단락을 넘어 복사되는 시나리오로, 연속 대형 표의 컨텍스트 보존을 위한 것입니다.
4. **표 행은 서로 중복하지 않음**: 행 수준 슬라이싱 자체는 비중복이며, R의 중복 개념과 다릅니다.

## 3. 입력과 출력

### 3.1 입력

`chunking_by_paragraph_semantic()`은 다음 입력을 받습니다:

| 파라미터 | 소스 | 설명 |
|---|---|---|
| `content` | `full_docs[doc_id].content` | 사이드카가 없을 때 폴백으로 사용하는 연결 병합 텍스트 |
| `blocks_path` | `full_docs[doc_id].lightrag_document_path` | `.blocks.jsonl` 경로, P 전략의 주요 입력 |
| `chunk_token_size` | `chunk_options.chunk_token_size` / `CHUNK_P_SIZE` | 목표 하드 상한 N; 기본값 `2000` |
| `chunk_overlap_token_size` | `CHUNK_P_OVERLAP_SIZE` / `chunk_overlap_token_size` | 동일 내용 행 내의 긴 본문 폴백 중복 및 표 연결 예산의 상한; 기본값 `100` |
| `tokenizer` | LightRAG가 이미 파싱한 토크나이저 | 모든 토큰 계산과 텍스트 중복 절단의 기준 |

P 전략은 `split_by_character` / `split_by_character_only`를 **허용하지 않습니다**. 정상 경로는 제목과 단락 구조로 구동되기 때문입니다.

### 3.2 `.blocks.jsonl` 규약

P 전략은 `type == "content"` 행만 처리합니다. 각 내용 행은 일반적으로 다음을 포함합니다:

- `content`: 제목 아래의 본문 텍스트. 일반 단락, `<table ... />` 태그, `<equation ... />` 수식, `<drawing ... />` 그래픽을 포함할 수 있습니다.
- `heading`: 현재 제목.
- `parent_headings`: 상위 제목 체인.
- `level`: 제목 수준 (1–9, 원래 개요 수준 0–8에 해당).
- `positions`: 원본 단락 위치 정보 (추적용).

native 파서의 `fixlevel=0` 모드는 "제목 아래의 본문이 하나의 기본 청크가 됨"을 보장하며, 파싱 중에 토큰 임계값 분할을 수행하지 않습니다. 표는 `content`에 삽입되면서 그대로 유지됩니다.

### 3.3 출력

최종 출력은 순서가 있는 청크 목록으로, 각 요소는 다음과 같습니다:

```python
{
    "tokens": int,                    # 실제 토큰 수 (병합 후 재측정)
    "content": str,                   # 청크 텍스트 (<table> 태그 포함 가능)
    "chunk_order_index": int,         # 청크 순서 인덱스
    "heading": str,                   # 분할 후 [part n] 접미사가 추가됨
    "parent_headings": list[str],     # 상위 제목 체인; 접미사 추가 안 됨
    "level": int,                     # 제목 수준
}
```

내부적으로 구현은 분할과 병합을 보조하기 위해 `paragraphs`, `table_chunk_role`, `uuid`, `uuid_end`, `type` 같은 필드를 임시로 사용하지만, **최종 출력에는 나타나지 않습니다**.

### 3.4 `[part n]` 접미사 규칙

- 동일한 원본 `.blocks.jsonl` 내용 행이 여러 슬라이스로 분할될 때, 모든 슬라이스의 `heading` 필드에 `[part 1]`, `[part 2]` … 가 추가됩니다.
- 분할되지 않는 내용 행은 원본 제목을 그대로 유지합니다.
- `parent_headings`에는 어떤 접미사도 추가되지 않습니다.
- 번호는 **각 원본 내용 행 내에서 독립적으로 재설정**됩니다.
- 레거시 `[表格片段N]`("표 조각 N") 접미사는 일괄적으로 `[part n]`으로 교체됩니다.

## 4. 핵심 임계값

P 전략의 임계값은 고정 상수가 아니라 `chunk_token_size`(N으로 표기)에서 동적으로 파생됩니다:

| 이름 | 공식 | N = 2000일 때 값 | 기술적 의미 |
|---|---|---:|---|
| `target_max` | N | 2000 | 텍스트 청크의 하드 상한 |
| `target_ideal` | 0.75 × N | 1500 | 텍스트 청크의 이상적 목표; 이 값 이상에 도달한 청크는 일반 동등 병합에 참여하지 않음 |
| `table_max` | 0.625 × N | 1250 | 표 슬라이싱을 촉발하는 임계값 |
| `table_ideal` | 0.375 × N | 750 | 표 슬라이스의 이상적 크기 |
| `table_min_last` | 0.32 × `table_max` | 400 | 마지막 슬라이스 역흡수 임계값 (마지막 슬라이스가 더 작고 병합 가능하면 이전 슬라이스로 역흡수) |
| `small_tail_threshold` | 0.125 × N | 250 | 꼬리 조각 흡수 임계값 |
| `max_anchor_candidate_length` | 고정값 | 100자 | 긴 청크 분할의 후보 앵커 단락 길이 상한 |

비례 제약 관계: `table_max < target_ideal < target_max`, `table_ideal < table_max`. 이 비율은 감사 모드(대형 청크 8000, 소형 표 5000, 이상적 표 3000, 표 꼬리 1600)의 경험적 값에서 유래하며, 현재 `chunk_token_size`에 비례 스케일링됩니다.

## 5. Stage A: 제목 수준 기본 청크

제목 인식은 native 파서가 수행합니다; **P 청커 자체는 docx 본문을 스캔하거나 제목 스타일을 판단하지 않습니다**.

`fixlevel=0` 모드에서 native 파서는:

1. `styles.xml`을 읽고, `<w:basedOn>`을 통해 스타일 상속 체인을 구성하여 유효한 `<w:outlineLvl>`을 추적합니다.
2. `document.xml`의 단락을 반복하면서 상속 체인을 따라 개요 수준을 결정; 원래 개요 수준 0–8은 내부 `level` 1–9로 매핑됩니다.
3. `current_heading_stack`을 유지하고, 새 제목이 나타날 때 현재 수준보다 얕지 않은 이전 제목을 지우고 `parent_headings`를 계산합니다.
4. 표, 수식, 그래픽을 단일 행 태그(`<table id="..." format="json">...</table>` 등)로 추출하여 해당 사이드카에 씁니다.
5. 인식 가능한 모든 제목이 기본 청크 경계를 촉발하며; 토큰 임계값 분할은 수행 **하지 않습니다**.

P 청커는 `.blocks.jsonl`을 직접 읽고, 각 내용 행을 후속 Stage B/C의 독립적인 처리 단위로 취급합니다. 이는 `[part n]` 번호 부여가 원본 내용 행별로 독립적으로 재설정됨을 의미합니다.

## 6. Stage B: 초과 크기 표의 행 경계 슬라이싱

Stage B는 토큰 수가 `table_max`를 초과하는 표만 처리합니다. 목표는 **단순히 표를 분할하는 것이 아니라**, 행 경계 우선 분할을 기반으로 표 경계 컨텍스트를 보존하는 것입니다.

### 6.1 행 경계 우선 슬라이싱

- `format="json"`: 최상위 JSON 행 배열로 슬라이스.
- `format="html"`: `<tr>...</tr>` 행으로 슬라이스.
- 명시적으로 태그되지 않았지만 JSON / HTML로 감지 가능한 표도 동일한 규칙 적용.

슬라이싱 전에 `<table {attrs}></table>` 래퍼 토큰 비용이 미리 공제되어, 각 재포장 슬라이스가 가능한 한 `table_max` 미만을 유지합니다. 각 슬라이스는 다운스트림 파싱을 위해 유효한 `<table>` 태그로 재포장됩니다.

### 6.2 행 수준 재귀적 재슬라이싱

행 부분 집합이 재포장 후에도 `table_max`를 초과하면, 해당 행 부분 집합 내에서 추가 세분화가 수행됩니다. **슬라이싱이 단일 행으로 수렴했는데 그 행 자체가 한계를 초과할 때만 문자 수준 분할로 저하됩니다**. 이 메커니즘은 행 경계로 표현 가능한 표 내용에 대해 가능한 한 많은 유효한 표 구조를 유지합니다.

### 6.3 마지막 슬라이스 역흡수

마지막 표 슬라이스의 토큰 수가 `table_min_last` 미만이고, 이전 슬라이스와 병합한 결과가 `table_max`를 초과하지 않으면, 마지막 슬라이스를 이전 슬라이스로 역흡수하여 쓸모없는 짧은 표 청크를 줄입니다.

### 6.4 표 슬라이스 역할과 물리적 결합

각 표 슬라이스에는 내부 필드 `table_chunk_role`이 부여되며, 역할에 따라 주변 단락과의 결합 방식이 결정됩니다:

| 역할 | 의미 | 결합 전략 |
|---|---|---|
| `first` | 원본 표의 첫 번째 슬라이스 | 표의 **앞 설명**이 첫 번째 슬라이스와 같은 청크에 들어가도록 현재 누적 청크의 끝에 추가 |
| `middle` | 원본 표의 중간 슬라이스 | 관련 없는 본문과 병합되지 않도록 독립적으로 출력 |
| `last` | 원본 표의 마지막 슬라이스 | **다음 설명**이 마지막 슬라이스 뒤에 자동으로 추가되도록 새 누적 청크의 시작점으로 사용 |
| `none` | 비표 슬라이스 또는 그대로 유지된 전체 표 | 일반 텍스트 청크로 처리 |

`table_chunk_role`은 최종 출력에 남지 않는 내부 필드이지만, **Stage D에서 병합 제약으로 계속 사용됩니다**(§9.1 참조).

## 7. Stage B.1: 연속 대형 표 사이 연결 텍스트의 양방향 중복

동일 원본 내용 행에서 "대형 표 A, 짧은 연결 텍스트, 대형 표 B" 패턴이 나타나고 두 표가 모두 분할된 경우, 연결 텍스트는 컨텍스트 예산에 따라 양방향으로 분배됩니다:

1. 연결 텍스트를 토큰으로 인코딩.
2. 좌측 예산 계산: `prev_budget = min(chunk_overlap_token_size, target_max - 왼쪽 마지막 슬라이스의 현재 토큰 수)`.
3. 우측 예산 계산: `next_budget = min(chunk_overlap_token_size, target_max - 오른쪽 첫 번째 슬라이스의 현재 토큰 수)`.
4. **연결 텍스트 길이가 두 예산을 초과하지 않으면**: 좌우 표 경계 청크 모두 **완전한 연결 텍스트**를 포함.
5. **연결 텍스트가 더 길면**: 접두부는 왼쪽 마지막 슬라이스 청크에, 접미부는 오른쪽 첫 번째 슬라이스 청크에 들어가며; 두 예산을 모두 초과하는 중간 부분은 독립적인 일반 텍스트 청크가 됩니다.

각 단측 예산은 연결 텍스트가 전체 청크를 지배하지 못하도록 추가로 `chunk_token_size / 2`로 제한됩니다.

일반 인접 청크 중복과의 차이점:

- 일반 중복은 경계 유형에 관계없이 순서대로 문자나 토큰을 복사합니다.
- B.1 메커니즘은 표 슬라이스 역할에 의해 트리거되며, 연결 텍스트를 왼쪽 표의 후속 컨텍스트이자 오른쪽 표의 선행 컨텍스트로 취급하여, 연결 설명이 한쪽에만 할당되거나 분리되어 회상하기 어렵게 되는 것을 방지합니다.

## 8. Stage C: 앵커 기반 긴 텍스트 청크 재분할

Stage C는 Stage B 후에도 `target_max`를 초과하는 내용 청크를 처리합니다.

### 8.1 짧은 단락 앵커

내용을 단락으로 복원한 후, 다음을 모두 만족하는 단락을 후보 앵커로 선택합니다:

- 단락이 표가 아님 (`<table`로 시작하지 않음).
- 단락 텍스트 길이가 `max_anchor_candidate_length`(100자)를 초과하지 않음.
- 청크의 첫 번째 단락이 아님 (비수렴 재귀를 피하기 위해).

### 8.2 균형 잡힌 앵커 선택

목표 하위 청크 수를 기반으로 이상적인 분할 위치를 계산하고, 후보 중에서 각 이상적 위치에 가장 가까운 앵커를 선택합니다. 선택된 앵커는 **다음 하위 청크의 새 `heading`으로 승격**되며, 원본 제목은 그 하위 청크의 `parent_headings`에 기록됩니다.

### 8.3 앵커 없을 때 폴백

적합한 앵커가 없는 경우:

1. **표 우선**: 청크 내에 초과 크기 표가 여전히 존재하면 Stage B의 행 경계 슬라이싱을 우선합니다.
2. **탐욕적 패킹**: 남은 텍스트를 단락 단위로 `target_max`에 접근하게 탐욕적으로 패킹합니다.
3. **재귀적 문자 분할**: 지나치게 긴 단일 일반 텍스트 단락은 R 전략(`chunking_by_recursive_character`)으로 폴백하며, 인접 텍스트 슬라이스 간 연속성을 위해 `chunk_overlap_token_size`를 사용합니다.

앵커 없음 폴백 경로는 알고리즘이 **내용을 폐기하지 않고** 사용자 설정 청크 크기 상한을 최대한 준수하도록 보장합니다.

## 9. Stage D: 계층 인식 2단계 병합

Stage D는 세분화된 섹션 시나리오에서 "청크가 너무 작음"과 "토픽 간 오염" 사이의 긴장을 해소합니다. 핵심 아이디어는 **깊은 수준에서 얕은 수준으로** 처리하며, 먼저 같은 수준에서 작은 청크를 병합하고, 그 후 얕은 청크가 깊은 청크를 흡수하도록 허용하면서, 크기 제약, 표 슬라이스 역할 제약, 제목 경로 제약을 도입합니다.

### 9.1 D.0 병합 제약 (모든 병합이 충족해야 함)

1. **크기 제약**: 병합 후 실제 텍스트 토큰 수가 `target_max`를 초과하지 않음; 이미 `target_ideal`에 도달한 청크는 원칙적으로 일반 동등 병합에 계속 참여하지 않음.
2. **역할 제약**: `middle` 표 슬라이스는 독립으로 잠금; `first`와 `last`는 방향성 있게 병합에 참여하여 표 경계 컨텍스트가 잘못 흡수되는 것을 방지.
3. **수준 제약**: 동등 병합은 동일 `level` 간에 발생; 교차 수준 흡수는 얕은 것이 깊은 것을 흡수하는 것만 허용, **깊은 것이 얕은 것을 역으로 흡수하는 것은 금지**.
4. **상위 제목 경로 일관성 제약**: 인접 청크의 `parent_headings`가 동일하거나, 같은 상위 제목 경로로 제한된 연속 범위 내에 있어야 함. 이것이 토픽 간 오염을 방지하는 핵심입니다.

### 9.2 D.1 1단계: 동등 병합

현재 수준의 인접 청크에 대해, 둘 다 `target_ideal` 미만이고 위의 제약을 충족하면 하나의 청크로 병합합니다.

표 슬라이스 역할의 방향성 규칙:

| 청크 역할 | 다음 청크를 전방 흡수 가능 | 이전 청크에 흡수 가능 |
|---|:-:|:-:|
| `none` | 예 | 예 |
| `first` | 예 | 아니오 |
| `middle` | 아니오 | 아니오 |
| `last` | 아니오 | 예 |

### 9.3 D.2 일괄 꼬리 흡수

`target_ideal`에 도달한 청크 뒤에 작은 동등 청크의 연속이 있고, 그 연속의 총 토큰 수가 `small_tail_threshold` 미만이며 실제 병합된 토큰 수가 `target_max`를 초과하지 않으면, **그 연속을 한 번에 흡수합니다**. `middle` 표 슬라이스를 만나면 중지.

### 9.4 D.3 2단계: 교차 수준 흡수

1단계 후에도 포화되지 않은 작은 청크에 대해 교차 수준 병합을 시도하되, 얕은 것이 깊은 것을 흡수하는 것만 허용합니다:

- 현재 청크가 다음 청크보다 얕으면, 현재 청크가 다음을 전방 흡수할 수 있습니다.
- 현재 청크가 이전 청크보다 깊으면, 이전 얕은 청크가 현재를 흡수할 수 있습니다.
- 역방향 병합은 금지.
- 교차 수준 단계에서는 `last` 역할도 전방 흡수 허용; `middle`은 여전히 병합에 참여하지 않음.

### 9.5 D.4 병합 후 실제 토큰 재측정

병합 시 줄바꿈 연결자가 삽입되기 때문에, 청크별 토큰 합산은 병합 결과를 과소평가할 수 있습니다. **각 병합을 확정하기 전에 실제 연결 텍스트를 재토크나이징해야 하며**, `target_max`를 초과하지 않음을 확인한 후에만 병합을 확정합니다.

병합 후 메인 청크의 `heading`이 유지됩니다. 여러 part 슬라이스가 병합된 경우, 최종 제목은 메인 청크의 part 접미사를 유지하며, **여러 part 태그를 추가로 연결하지 않습니다**.

## 10. 폴백 및 저하 경로

P 전략에는 여러 겹의 폴백 보호가 있습니다:

| 트리거 | 저하 동작 |
|---|---|
| `blocks_path` 없음, 읽기 불가, 유효한 내용 행 없음 | 완전히 `chunking_by_recursive_character()`로 폴백하여 파싱된 `chunk_overlap_token_size` 전달 |
| Stage B가 표의 JSON / HTML 구조를 식별할 수 없음 | 해당 표에 R 전략의 문자 분할 사용 |
| Stage B에서 단일 행 표 자체가 `table_max` 초과 | 해당 단일 행에 R 전략의 문자 분할 사용 |
| Stage C에서 적합한 짧은 단락 앵커가 없는 긴 청크 | 표 우선 → 탐욕적 패킹 → 단일 단락이 너무 길면 R 문자 분할로 폴백 |

**중요**: 전체 폴백 후에는 제목 계층, 표 역할, 양방향 연결 텍스트 중복 같은 기능을 더 이상 사용할 수 없습니다; 그러나 문서가 구조화된 사이드카 없음으로 인해 자동으로 폐기되지 않고 검색 청크를 생성하도록 보장합니다.

## 11. 설정

| 설정 | 기본값 | 설명 |
|---|---|---|
| `CHUNK_P_SIZE` | `2000` (미설정 시 `DEFAULT_CHUNK_P_SIZE` 사용; `CHUNK_SIZE`로 **폴백하지 않음**) | P 전용 `chunk_token_size`; 단락 시맨틱 병합은 전역 기본값보다 더 높은 상한을 필요로 하므로, `CHUNK_SIZE`로 폴백하지 않는 독립 기본값 사용 |
| `CHUNK_P_OVERLAP_SIZE` | 미설정 (`CHUNK_OVERLAP_SIZE`로 폴백) | P 전용 중복; 동일 내용 행 내의 긴 본문 폴백 중복과 표 연결 예산에만 영향. 표 행 수준 슬라이스가 중복되지는 **않음** |
| `CHUNK_OVERLAP_SIZE` / `LightRAG(chunk_overlap_token_size=…)` | `100` | P 전용 중복이 설정되지 않았을 때의 전역 폴백 |

설정 구문, 우선순위 체인 및 `addon_params["chunker"]`를 통한 런타임 재정의에 대해서는 [FileProcessingPipeline-ko.md](FileProcessingPipeline-ko.md) §3을 참조하세요.

P를 활성화하는 일반적인 `LIGHTRAG_PARSER` 설정:

```bash
LIGHTRAG_PARSER=docx:native-P,*:legacy-R
CHUNK_P_SIZE=2000
CHUNK_P_OVERLAP_SIZE=100
```

또는 단일 파일에 재정의:

```text
my-proposal.[native-P].docx
```

## 12. 청킹 결과 검증

### 12.1 사이드카가 생성되었는지 확인

native 파서가 `.blocks.jsonl`을 성공적으로 생성했는지 확인:

```bash
ls -l INPUT/__parsed__/<doc>.docx.parsed/<doc>.blocks.jsonl
```

파일이 없거나 비어 있으면 P 전략은 R로 완전히 폴백하고 P의 혜택을 전혀 얻지 못합니다. 일반적인 원인:

- `LIGHTRAG_PARSER=docx:native-...`가 설정되지 않음.
- 파싱 실패 (`pipeline_status`의 오류 항목 참조).
- 문서가 DOCX가 아님 (다른 형식은 P를 지원하지 않음).

### 12.2 blocks.jsonl 내용 검사

각 행은 JSON이며; `type == "content"`를 필터링하고 heading / level / parent_headings가 기대에 맞는지 검사:

```bash
jq -c 'select(.type=="content") | {level, heading, parent_headings}' \
   INPUT/__parsed__/<doc>.docx.parsed/<doc>.blocks.jsonl | head
```

대부분의 제목이 비어 있거나 수준이 비정상이면 native 파서가 제목 스타일을 올바르게 인식하지 못한 것으로, P의 계층 병합과 앵커 승격 모두 실패할 것입니다.

### 12.3 최종 청크 검사

`text_chunks` 스토리지의 청크 메타데이터 조회:

```bash
jq '.[] | {heading, level, tokens, parent_headings}' \
   rag_storage/kv_store_text_chunks.json | head -30
```

다음을 확인해야 합니다:

- 대형 표 주변 청크의 제목은 일반적으로 `[part 1]` / `[part n]`에 해당 (Stage B 분할이 발생했음을 나타냄).
- 세분화된 조항들은 `target_ideal`에 가까운 청크로 병합됨 (Stage D가 효과를 발휘했음을 나타냄).
- `parent_headings`는 서로 다른 섹션 간 경계에서 바뀌고 동일 섹션 내에서는 안정적으로 유지됨.

### 12.4 청크 크기 분포 확인

이상적인 분포: 대부분의 청크가 `[target_ideal, target_max]` 범위에 있음 (N=2000일 때 약 1500–2000 토큰); 눈에 띄게 작은 청크는 보통 `middle` 표 슬라이스(독립으로 잠금)이거나 섹션 경계의 꼬리 청크입니다.

`small_tail_threshold` 미만의 꼬리 청크가 많이 나타나면 가능한 원인:

- 상위 제목 경로 일관성 제약이 너무 엄격함 (서로 다른 `parent_headings`를 가진 인접 작은 청크들이 병합 불가).
- 많은 `middle` 표 슬라이스가 쌓임 (표 자체가 매우 큼).

## 13. 문제 해결

### 13.1 P가 적용되지 않음; 출력이 R과 일치

다음 순서로 조사:

1. `full_docs[doc_id].process_options`에 `P`가 있는가?
2. `full_docs[doc_id].parse_format`이 `lightrag`인가? `raw`이면 레거시 경로에 있어 P가 자동으로 R로 폴백됨.
3. `lightrag_document_path`가 가리키는 `.blocks.jsonl`이 존재하고 비어 있지 않은가?
4. 로그에 `paragraph_semantic ... fallback to recursive_character` 메시지가 있는가?

### 13.2 표가 산재함; 앞뒤 설명이 분리됨

- 표가 실제로 `<table format="json">` 또는 `<table format="html">`로 인식되었는지 확인 (`.blocks.jsonl` 참조). 인식되지 않은 형식의 표는 문자 분할만 가능하고 Stage B의 역할 메커니즘을 트리거할 수 없습니다.
- 표의 토큰 수가 실제로 `table_max`를 초과하는지 확인. 임계값 미만의 표는 그대로 유지되어 first/middle/last 슬라이싱을 절대 트리거하지 않습니다.
- 연속 대형 표의 경우, 두 표 사이의 연결 텍스트가 **동일 내용 행** 내에 있는지 확인 — 내용 행을 넘는 연결은 B.1 양방향 중복에 참여하지 않습니다.

### 13.3 세분화된 조항이 병합되지 않음

- 인접 조항의 `parent_headings`가 동일한지 확인: 상위 제목 경로 일관성 제약은 토픽 간 병합을 방지합니다.
- `level`이 동일한지 확인: 동등 병합은 동일 `level`을 요구; 교차 수준 흡수는 얕은 것이 깊은 것을 흡수하는 것만 허용합니다.
- 중간에 `middle` 표 슬라이스가 삽입되어 있는지 확인: 이는 일괄 꼬리 흡수를 차단합니다.

### 13.4 단일 청크가 `target_max` 초과

일반적으로 Stage D의 실제 토큰 재측정이 초과 크기 병합을 거부하지만, 다음 시나리오에서는 초과 크기 청크가 여전히 발생할 수 있습니다:

- 단일 행 표 자체가 `target_max`를 초과하고 분할할 앵커가 없음; 결국 R 문자 분할을 거치지만 단일 청크가 여전히 한계를 초과합니다.
- `enforce_chunk_token_limit_before_embedding`이 임베딩 전에 최종 하드 절단을 수행; 다운스트림은 실제로 초과 크기 청크를 벡터 스토어에 임베딩하지 않습니다.

### 13.5 비정상적인 `[part n]` 접미사

- 여러 슬라이스가 동일한 원본 내용 행에서 왔지만 `[part 1]`만 하나 보임: Stage D에서 병합되었는지 확인 — 병합 후 메인 청크의 part 접미사가 유지되고 여러 part 태그는 연결되지 않습니다.
- 레거시 `[表格片段N]` 접미사가 나타남: 이전 청커가 출력한 데이터를 나타냄; 새 버전은 `[part n]`으로 표준화되었으며, 재청킹이 필요합니다.

### 13.6 로그 키워드

P 전략 관련 로그 키워드 (`grep` 기반 문제 해결용):

- `paragraph_semantic` — 모듈 진입점
- `fallback to recursive_character` — 전체 또는 단일 단락 저하
- `table_chunk_role` — 표 역할 관련
- `bridge` — Stage B.1 연결 텍스트 처리
- `anchor` — Stage C 앵커 선택
